## 六、任务检查规范（关键！）

### 6.1 检查清单

```yaml
检查步骤:
  1. 文件存在性检查:
    - Entity 是否存在
    - Mapper 是否存在
    - Service/ServiceImpl 是否存在
    - Controller 是否存在
    - DTO 是否存在

  2. 代码完整性检查:
    - Service 是否包含完整 CRUD 方法:
      - pageQuery() 分页查询
      - getDetail() 获取详情
      - create() 创建
      - update() 更新
      - delete() 删除
    - Controller 是否包含对应 API 端点

  3. 实现逻辑审查（关键！）:
    - 必须阅读代码内容，检查逻辑是否正确
    - 检查方法是否有实际实现，不是空方法
    - 检查业务逻辑是否符合设计文档要求
    - 检查事务注解是否正确使用

  4. 编译验证:
    - 后端: mvn compile
    - 前端: npm run build
```

### 6.2 检查命令

```bash
# 后端编译验证
cd backend && mvn compile -q 2>&1 | tail -10

# 前端编译验证
cd frontend && npm run build 2>&1 | grep -c "error TS"

# 检查文件存在
ls backend/src/main/java/com/duoduo/jxc/entity/Xxx.java
ls backend/src/main/java/com/duoduo/jxc/service/impl/XxxServiceImpl.java

# 检查代码内容
cat backend/src/main/java/com/duoduo/jxc/service/impl/XxxServiceImpl.java
```

### 6.3 警告

> **严重警告**：只检查文件名/方法名而不检查实现逻辑，将被开除！
> 管理者必须对代码质量负责，不能敷衍了事。

## 七、通知规范

```yaml
通知原则:
  1. 已通知的任务不重复通知:
    - 任务下达后，只需等待执行
    - 不重复发送相同的任务指令

  2. 定时检查状态:
    - 用户发送"检查状态"时检查
    - 检查文件是否存在 + 代码是否完整
    - 直到任务完成为止

  3. 发现问题再通知:
    - 检查发现问题时，通知对应AI修复
    - 修复后继续检查，直到通过
```
